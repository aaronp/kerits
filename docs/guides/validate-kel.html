Validate a KEL

Verify SAID integrity, structural fields, key-chain continuity, and full chain validity.

KERI's security model is built on the assumption that verifiers independently validate every event in a KEL before trusting any key state derived from it. Unlike PKI, there is no trusted third party: validity is a property of the event log itself, derivable from cryptographic rules alone.

@kerits/core exposes granular validation helpers so you can apply exactly the checks you need, plus a high-level validateKelChain function that runs all checks in sequence and returns structured, per-event results.

Check SAID integrity

validateEventSaid recomputes an event's SAID from its body and compares it to the d field. A mismatch means the event was tampered with or malformed.

Check required fields

validateRequiredFields verifies that every field mandated by the event type (icp, rot, ixn, dip, drt) is present. This catches truncated or incorrectly constructed events before you attempt signature verification.

Required fields by event type:

EventRequired fields

Validate key-chain continuity

validateKeyChain checks that a rotation event's current keys (k) satisfy the previous establishment event's next-key commitments (n). Each key must hash (via BLAKE3-256) to one of the digests listed in n, and enough of those digests must be revealed to satisfy the nt threshold.

Full chain validation

validateKelChain runs all checks — SAID integrity, required fields, prior-event linkage (p field), signature verification, threshold satisfaction, key-chain continuity, and delegation approval — for every event in the array. It returns a RichValidationResult with per-event details.
